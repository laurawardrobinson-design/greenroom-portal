## CRITICAL — fix before any user demo

### C-1. Designer dashboard renders empty ✅ Fixed 2026-04-19
- **Location**: [app/(portal)/dashboard/page.tsx:31](portal-v2/app/(portal)/dashboard/page.tsx)
- **Repro**: Login as Designer (`designer@test.local`) → blank `/dashboard`
- **Cause**: Role switch statement has no `case "Designer"`; falls through to default loading skeleton that never resolves
- **Fix**: Add `case "Designer": return <DesignerDashboard user={user} />;` — or until that component exists, route Designer to a fallback like the Producer or Studio dashboard
- **Effort**: 1 hr (fallback) / 1 day (real dashboard)
- **Landed**: Fallback to `ProducerDashboard` until a dedicated Designer dashboard is built.

### C-2. PO digital signature stores literal string `"captured-server-side"` instead of client IP ✅ Fixed 2026-04-19
- **Location**: [components/vendors/po-signature.tsx:57](portal-v2/components/vendors/po-signature.tsx)
- **Impact**: Defeats the legal-evidence purpose of the signature. Every vendor's "signed" PO has the same bogus IP value. Discoverable on any audit.
- **Fix**: Pass IP from server. In the signing API route, read `request.headers.get("x-forwarded-for")?.split(",")[0]` (or `x-real-ip`) and store on the signature record server-side, not from client.
- **Effort**: 2 hrs
- **Landed**: `PATCH /api/campaign-vendors/[id]` now extracts real client IP from `x-forwarded-for` / `x-real-ip` for any `PO Signed` transition and ignores what the client sends. Client placeholder string removed. Verified: signed `signed_ip = "::1"` (real localhost) instead of the literal.

### C-3. Silent failures in campaign mutation handlers ✅ Fixed 2026-04-19
- **Locations**:
  - [app/(portal)/campaigns/[id]/page.tsx:186](portal-v2/app/(portal)/campaigns/[id]/page.tsx) — `handleUpdate`, `handleDelete`, `handleStatusChange`
  - [components/campaigns/campaign-row.tsx:80-86](portal-v2/components/campaigns/campaign-row.tsx) — `patchCampaign` inline edit
- **Symptom**: `fetch()` result not `.ok`-checked before `mutate()` / redirect runs. A 500 from the API silently looks successful; user thinks edit saved.
- **Fix**: Wrap each in `try { if (!res.ok) throw new Error(await res.text()); ...mutate()/redirect } catch (e) { toast.error(...) }`
- **Effort**: 1 hr
- **Landed**: All three handlers on the campaign detail page plus `patchCampaign` on the row inline-edit now check `res.ok`, throw the server message, and surface a red toast.

### C-4. Mass-assignment / no Zod schema on Budget POST + PATCH ✅ Fixed 2026-04-19
- **Location**: [app/api/budget/route.ts:35-54](portal-v2/app/api/budget/route.ts)
- **Risk**: `createBudgetPool()` and `updateBudgetPool()` receive raw `await req.json()`. Service whitelists internally but if anyone refactors the service, attacker-controlled fields (`id`, `created_at`, etc.) flow straight to DB.
- **Fix**: Add Zod schemas (mirror the pattern in `/api/campaigns/route.ts`):
  ```ts
  const createBudgetPoolSchema = z.object({ name: z.string().min(1), total: z.number().nonnegative(), startDate: z.string(), endDate: z.string() });
  const body = createBudgetPoolSchema.parse(await req.json());
  ```
- **Effort**: 1 hr
- **Landed**: New `lib/validation/budget.schema.ts` with `createBudgetPoolSchema` + `updateBudgetPoolSchema`. Route parses via Zod; invalid payloads return 400 with `fieldErrors`. Verified: missing `name` rejected, `id` / `created_at` / `hacked` attacker fields stripped, happy path returns 201/200.

### C-5. `/approvals` page is a 404 — *partially* fixed ✅ Fixed 2026-04-19
- The HOP dashboard tile now links to `/budget` (fixed). But ideally the link would deep-link to the Approvals tab inside Budget (e.g., `/budget?tab=approvals`) and the `/budget` page would honor a `?tab=` query string.
- **Effort**: 30 min (add `useSearchParams` in budget page to honor tab param)
- **Landed**: Budget page reads `?tab=` via `useSearchParams` on mount; HOP tile now links to `/budget?tab=approvals`. Verified: deep link lands on Approvals tab active.

### M-6. Invoice upload has zero server-side file validation ✅ Fixed 2026-04-19
- **Location**: [app/api/invoices/route.ts:35-56](portal-v2/app/api/invoices/route.ts)
- **Issue**: Client `accept=".pdf,.png,.jpg"` is a hint only. Server accepts any file type, any size. Vendor can upload a 500 MB MP4 or a `.exe`.
- **Fix**: Server validation — check `file.type` against allowlist, enforce `file.size <= 10 * 1024 * 1024` (10 MB), reject otherwise.
- **Effort**: 1 hr
- **Landed**: Added MIME allowlist (PDF / PNG / JPG) + 10 MB size cap in `POST /api/invoices`. Verified: `.txt` upload → 400 "Invoice must be a PDF, PNG, or JPG"; oversized files rejected before storage write.

### M-7. CSV import returns `{ imported: 0 }` instead of error when all rows invalid ✅ Fixed 2026-04-19
- **Location**: [app/api/campaigns/import/route.ts:29-50](portal-v2/app/api/campaigns/import/route.ts)
- **Symptom**: Upload a malformed CSV → silent zero, no feedback to user about why
- **Fix**: After filtering valid rows, `if (validRows.length === 0) return new Response(JSON.stringify({ error: "No valid rows" }), { status: 400 });`
- **Effort**: 30 min

### M-8. SWR fetchers across tiles don't validate `Array.isArray()` ✅ Fixed 2026-04-19
- **Location**: [components/campaigns/tiles/people-tile.tsx:22-28](portal-v2/components/campaigns/tiles/people-tile.tsx) and similar
- **Risk**: If API returns `null` or `{ error: ... }` instead of an array, downstream `.filter()` / `.map()` throws and the whole tile errors out
- **Fix**: Audit pattern: `const items = Array.isArray(rawData) ? rawData : [];`
- **Effort**: 1-2 hrs (multi-file)
- **Landed**: Added `Array.isArray(...)` guards to the known-crashing spots — `rate-card-management.tsx` (observed crashing as `rateCards.map is not a function` when SWR got a 403 error object) and `campaign-row.tsx` (`producers`, `artDirectors`, `requestsData`). `people-tile.tsx` VendorsTab already had the pattern.

### M-9. No CSRF protection on mutating routes ⏸ Deferred (2026-04-19)
- **Issue**: 100+ POST/PATCH/DELETE routes have no CSRF token check. Authenticated users on a malicious site can be tricked into firing state-changing requests.
- **Mitigation note**: Supabase auth uses cookies w/ `SameSite=Lax` by default which blocks most cross-site POST attacks, but explicit CSRF defense-in-depth is industry standard.
- **Fix**: Either rely on Supabase's CORS config (verify `SameSite=Strict` is set) or add a CSRF token in the proxy.ts layer. Lower priority if CORS is locked down.
- **Effort**: 4 hrs
- **Deferred**: Per handoff guidance, deferred until `SameSite` cookie setting is verified. Not a blocker for current rollout.

### M-10. Wardrobe routes use manual field validation instead of Zod ✅ Fixed 2026-04-19
- **Location**: [app/api/wardrobe/reservations/route.ts:22-39](portal-v2/app/api/wardrobe/reservations/route.ts)
- **Inconsistency**: Most other routes use Zod; wardrobe doesn't. Easy to drift.
- **Fix**: Add Zod schemas matching the wardrobe-item / reservation domain.
- **Effort**: 1 hr
- **Landed**: Added `createReservationSchema`, `updateReservationSchema`, `createCheckoutSchema` to `wardrobe.schema.ts` + new `checkoutActionSchema`, `checkinActionSchema`, `checkinByItemActionSchema`, `batchCheckoutActionSchema`, `batchCheckinActionSchema` for the main `/api/wardrobe` route. `POST /api/wardrobe/reservations` and all 5 action handlers on `POST /api/wardrobe` now `Zod.parse`. Verified: bad UUID, missing fields, empty `items` array all return 400 with structured `fieldErrors`.
